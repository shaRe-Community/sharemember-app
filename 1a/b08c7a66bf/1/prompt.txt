# QR Code In-Person Identity Verification — Implementation Plan

## Context

Peer vouching exists but requires the requestee to know the voucher's email upfront. The new flow enables **in-person verification**: the requestee shows a QR code on their phone, the voucher scans it, and the sharemember-app opens with a one-tap confirm.

**Key design insight**: An "open" VouchRequest (no specific target voucher) is created when the requestee presses the button. The QR code encodes a deep link (`${VITE_PUBLIC_URL}/vouch/{requestId}`). When the voucher scans and opens the link, the existing VouchPage loads — the backend auto-claims the request for the scanning user on `GET`, then the normal confirm flow proceeds. **Zero changes needed to VouchPage UI or confirm endpoint.**

## Backend — operator-ts

### 1. New repository methods (`src/sharemember/vouch.repository.ts`)

**`createOpenRequest(requesterKeycloakId, requestId)`** — VouchRequest with only `[:FROM]`, no `[:TO]`:
```cypher
MATCH (sm:ShareMember {keycloakId: $requesterKeycloakId})
CREATE (req:VouchRequest {id: $requestId, requestedAt: datetime(), status: 'pending', type: 'open'})
CREATE (req)-[:FROM]->(sm)
RETURN req, sm AS requestee
```

**`cancelOpenRequests(requesterKeycloakId)`** — cancel stale open requests (idempotent QR regeneration):
```cypher
MATCH (sm:ShareMember {keycloakId: $requesterKeycloakId})<-[:FROM]-(req:VouchRequest {status: 'pending', type: 'open'})
WHERE NOT (req)-[:TO]->()
SET req.status = 'cancelled'
```

**`getOpenRequest(requestId)`** — fetch unclaimed open request (no voucher filter):
```cypher
MATCH (req:VouchRequest {id: $requestId})-[:FROM]->(requestee:ShareMember)
WHERE NOT (req)-[:TO]->()
RETURN req, requestee
```

**`claimOpenRequest(requestId, voucherKeycloakId)`** — atomically assign voucher:
```cypher
MATCH (req:VouchRequest {id: $requestId, status: 'pending'})-[:FROM]->(requestee:ShareMember)
WHERE NOT (req)-[:TO]->()
MATCH (voucher:ShareMember {keycloakId: $voucherKeycloakId})
WHERE voucher <> requestee
CREATE (req)-[:TO]->(voucher)
RETURN req, requestee
```
(`WHERE voucher <> requestee` prevents self-vouch)

**`getRequestStatus(requestId, requesterKeycloakId)`** — for polling from requestee's QR screen:
```cypher
MATCH (sm:ShareMember {keycloakId: $requesterKeycloakId})<-[:FROM]-(req:VouchRequest {id: $requestId})
RETURN req.status AS status
```

### 2. Service changes (`src/sharemember/vouch.service.ts`)

**New: `createOpenRequest(keycloakId, eidStatus)`**
- Reject if already identified (400)
- Cancel existing open requests
- Create new open request, return DTO

**New: `getRequestStatus(requestId, keycloakId)`**
- Return `{ status }` for polling

**Modified: `getRequest(requestId, voucherKeycloakId, voucherEidStatus)`**
- Try targeted lookup first (existing path)
- If not found → try open request lookup → validate voucher is IDENTIFIED (403 if not) → auto-claim → return DTO
- If claim fails (self-vouch) → throw 400

### 3. Controller changes (`src/sharemember/vouch.controller.ts`)

**New endpoint: `POST /v2/sharemember/vouch-requests/open`** (201)
- No body needed. Returns `VouchRequestDto`
- **Route ordering**: declare before `:requestId` params to avoid NestJS treating "open" as a requestId

**New endpoint: `GET /v2/sharemember/vouch-requests/:requestId/status`** (200)
- Returns `{ status: string }` — for requestee polling
- Only works for the requestee (FROM relationship)

**Modified: `GET /v2/sharemember/vouch-requests/:requestId`**
- Pass `user.eid_status` to service for open-request validation

### 4. Tests (`src/sharemember/vouch.service.spec.ts`)

Add ~6 tests:
- createOpenRequest: happy path, already-identified rejection, cancels existing
- getRequest open fallback: auto-claims for IDENTIFIED voucher
- getRequest open fallback: rejects UN-IDENTIFIED voucher (403)
- getRequest open fallback: rejects self-vouch (400)

## Security

- **Self-vouch**: Cypher `WHERE voucher <> requestee` — impossible to bypass
- **IDENTIFIED-only**: Service checks `eid_status` before claiming
- **Max 3 vouches**: Unchanged, checked on confirm
- **UUID request IDs**: Unguessable — scanning QR is the only discovery path
- **One open request per user**: `cancelOpenRequests` ensures freshness
